## 1. Overview

Hiseduplay는 한국사 학습과 콘텐츠 탐색을 목적으로 만든 웹 프로젝트입니다. 단순히 정보를 나열하는 페이지가 아니라, 방문자가 역사 콘텐츠를 보고, 퀴즈를 풀고, 질문과 답변을 남길 수 있는 작은 학습 서비스를 목표로 했습니다.

이 프로젝트는 사학 전공자로서의 관심사와 웹 개발 학습을 결합해 만든 결과물입니다. 그래서 기능 구성도 한국사 콘텐츠를 중심으로 설계했습니다.

- 한국사 콘텐츠 소개
- 오늘의 역사 퀴즈
- 역사 관련 상품 페이지
- 회원가입 및 로그인
- Q&A 게시판

## 2. Problem / Background

처음 프로젝트를 만들 당시의 목표는 "한국사 콘텐츠를 웹 서비스 형태로 구성해보는 것"이었습니다. 하지만 시간이 지난 뒤 포트폴리오로 다시 점검해보니, 코드는 남아 있었지만 처음 보는 사람이 실행하거나 구조를 이해하기에는 부족한 점이 있었습니다.

주요 문제는 다음과 같았습니다.

- Django 설정 값과 시크릿 데이터가 코드에 직접 포함되어 있었습니다.
- 로컬 데이터베이스, 캐시, 바이트코드 같은 생성 파일이 Git 추적 대상에 섞여 있었습니다.
- 이전 Django 스캐폴딩과 실제 프로젝트가 함께 남아 있어 구조가 혼란스러웠습니다.
- 의존성 파일과 실행 절차가 명확하지 않아 다른 환경에서 재현하기 어려웠습니다.
- 회원 비밀번호가 안전하게 저장되지 않는 레거시 흐름이 있었습니다.
- Q&A 기능 일부에서 런타임 오류가 발생할 수 있었습니다.
- 기능이 동작하는지 확인할 수 있는 자동 테스트가 부족했습니다.
- README가 텍스트 중심으로만 남아 있어 포트폴리오로 보기에는 시각 자료와 맥락이 부족했습니다.

따라서 단순히 "예전에 만든 프로젝트"를 보관하는 것이 아니라, 실제로 실행 가능하고 설명 가능한 포트폴리오 프로젝트로 정리하는 것이 필요했습니다.

## 6. Technical Approach / Process

### 6.1 Security Cleanup

가장 먼저 보안 정리를 진행했습니다. 기존 프로젝트에는 Django `SECRET_KEY`, 데이터베이스 설정, 로컬 DB 파일, 캐시 파일처럼 Git에 남기면 안 되는 데이터가 섞여 있었습니다.

개선한 내용은 다음과 같습니다.

- `DJANGO_SECRET_KEY`를 코드에서 제거하고 환경 변수로 이동
- 데이터베이스 설정을 환경 변수 기반으로 변경
- `hiseduplay/.env.example` 추가
- 실제 로컬 값은 ignored 파일인 `hiseduplay/.env.local`에서 관리하도록 변경
- SQLite DB, `__pycache__`, `.pyc` 등 생성 파일 제거
- `.gitignore`에 로컬 환경 파일과 생성 파일 규칙 추가

이 작업을 통해 저장소에는 예시 설정만 남기고, 실제 민감 정보는 Git에 올라가지 않도록 정리했습니다.

### 6.2 Project Structure Cleanup

기존 저장소에는 실제로 사용할 `hiseduplay` 프로젝트 외에도 오래된 Django 스캐폴딩과 루트 레벨 파일이 함께 남아 있었습니다. 이는 처음 보는 사람이 어느 프로젝트가 실제 실행 대상인지 파악하기 어렵게 만들었습니다.

개선한 내용은 다음과 같습니다.

- 실제 Django 프로젝트인 `hiseduplay`만 남김
- 루트의 오래된 `manage.py` 제거
- 사용하지 않는 이전 프로젝트 디렉터리 제거
- 불필요한 플랫폼 설정 파일 제거
- README의 프로젝트 구조를 현재 구조에 맞게 정리

정리 후 프로젝트의 중심 구조는 다음처럼 단순해졌습니다.

```text
hiseduplay/
  manage.py
  config/
  member/
  merch/
  qna/
  static/
  templates/
```

### 6.3 Dependency and Run Setup

프로젝트를 다른 환경에서도 재현할 수 있도록 실행 절차를 문서화하고 의존성 파일을 추가했습니다.

개선한 내용은 다음과 같습니다.

- `requirements.txt` 추가
- `requirements-mysql.txt` 추가
- `.python-version` 추가
- Python 3.11 기준 명시
- SQLite 기반 로컬 실행 절차 추가
- MySQL 사용 시 필요한 선택 설정 분리
- `python-dotenv` 기반 환경 변수 로딩 추가
- `manage.py check`, `migrate`, `test` 검증 명령 문서화

로컬 실행 흐름은 다음처럼 정리했습니다.

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
python -m pip install -r requirements.txt
Copy-Item hiseduplay\.env.example hiseduplay\.env.local
Set-Location hiseduplay
python manage.py migrate
python manage.py runserver
```

### 6.4 Authentication Fixes

기존 로그인 흐름에는 평문 비밀번호를 직접 비교하는 레거시 구현이 있었습니다. 포트폴리오 관점에서도, 실제 웹 프로젝트 관점에서도 반드시 개선해야 하는 부분이었습니다.

개선한 내용은 다음과 같습니다.

- 새 회원가입 시 Django password hasher로 비밀번호 저장
- 로그인 시 `check_password` 기반 검증 적용
- 기존 평문 비밀번호 사용자는 로그인 성공 시 해시 비밀번호로 자동 업그레이드
- 세션 기반 로그인 흐름 유지
- 인증 관련 테스트 추가

이 방식은 기존 데이터를 한 번에 모두 마이그레이션하지 않아도 되면서, 사용자가 정상 로그인하는 시점에 점진적으로 안전한 저장 방식으로 전환할 수 있다는 장점이 있습니다.

### 6.5 Runtime Bug Fixes

Q&A 기능에서 질문과 답변 작성 시 런타임 오류가 발생할 수 있는 흐름을 수정했습니다.

개선한 내용은 다음과 같습니다.

- 질문 상세 조회 시 명시적인 객체 조회 사용
- 존재하지 않는 질문에 대한 처리 개선
- 답변 생성 시 잘못된 필드 참조 수정
- 로그인되지 않은 사용자의 작성 흐름 처리 개선
- broad exception 사용을 줄이고 예측 가능한 리다이렉트 흐름 적용

이 작업으로 Q&A 기능이 단순히 화면만 존재하는 상태가 아니라, 실제로 질문 작성, 상세 조회, 답변 작성 흐름을 안정적으로 수행할 수 있게 되었습니다.

### 6.6 Django Best Practices

프로젝트 규모에 맞춰 과도한 구조 변경은 피하고, 기존 코드의 형태를 존중하면서 유지보수성을 높이는 방향으로 개선했습니다.

적용한 내용은 다음과 같습니다.

- settings에서 환경 변수 기반 설정 사용
- 로컬 설정 파일과 예시 설정 파일 분리
- URL name 정리
- 명시적인 리다이렉트와 객체 조회 사용
- 테스트 가능한 단위로 기능 동작 검증
- README에 실행, 환경 변수, 테스트 절차 문서화

### 6.7 Focused Tests

핵심 동작이 다시 깨지지 않도록 집중 테스트를 추가했습니다.

테스트 범위는 다음과 같습니다.

- 회원가입 시 비밀번호 해싱 확인
- 로그인 성공 및 실패 흐름 확인
- 기존 평문 비밀번호 로그인 후 해시 업그레이드 확인
- Q&A 질문 작성 확인
- Q&A 답변 작성 확인
- 존재하지 않는 질문 처리 확인
- merch 페이지 렌더링 확인
- quiz 페이지 렌더링 확인

현재 유지보수 버전은 총 11개의 집중 테스트를 통과하도록 정리되었습니다.

## 7. Key Decisions

### Decision 1. 기존 커스텀 Member 모델을 유지했다

처음부터 Django 기본 `User` 모델로 전면 전환할 수도 있었지만, 이 프로젝트는 이미 레거시 `MEMBER` 테이블과 커스텀 로그인 흐름을 가지고 있었습니다. 포트폴리오 유지보수 범위에서는 전체 인증 시스템을 갈아엎기보다, 기존 구조를 유지하면서 가장 위험한 비밀번호 저장 방식을 먼저 개선하는 편이 더 적절하다고 판단했습니다.

### Decision 2. 평문 비밀번호는 로그인 시점에 점진적으로 업그레이드했다

기존 사용자의 평문 비밀번호를 강제로 일괄 변경하면 데이터 손실이나 로그인 실패가 발생할 수 있습니다. 그래서 로그인에 성공한 경우에만 해당 비밀번호를 Django 해시 값으로 바꾸는 방식을 선택했습니다.

이 방식의 장점은 다음과 같습니다.

- 기존 계정의 로그인 흐름을 유지할 수 있음
- 새 계정은 즉시 안전한 방식으로 저장됨
- 기존 계정도 사용 시점에 점진적으로 안전해짐

### Decision 3. SQLite를 기본 로컬 DB로 두고 MySQL은 선택 설정으로 분리했다

포트폴리오를 보는 사람이 프로젝트를 쉽게 실행하려면 별도 DB 서버가 없어도 동작해야 합니다. 따라서 기본 로컬 개발은 SQLite로 실행 가능하게 두고, MySQL은 `requirements-mysql.txt`와 환경 변수 설정으로 분리했습니다.

### Decision 4. 테스트는 넓게보다 위험한 흐름 중심으로 추가했다

이 프로젝트의 가장 중요한 리스크는 인증과 Q&A 런타임 동작이었습니다. 따라서 테스트는 모든 화면을 포괄적으로 커버하기보다, 실제로 깨지면 서비스 품질에 영향이 큰 흐름을 중심으로 추가했습니다.

## 10. Troubleshooting

### 문제 1. 실제 프로젝트와 오래된 스캐폴딩이 함께 남아 있었다

저장소에 여러 Django 구조가 섞여 있어 실행 진입점을 파악하기 어려웠습니다. 실제로 사용하는 `hiseduplay`만 남기고 나머지는 제거해 구조를 단순화했습니다.

### 문제 2. 시크릿과 로컬 설정이 코드에 섞여 있었다

`SECRET_KEY`와 DB 설정이 코드에 직접 들어가 있으면 저장소가 private이어도 안전하지 않습니다. 설정 값을 환경 변수로 분리하고 `.env.example`에는 예시 값만 남겼습니다.

### 문제 3. 기존 회원 비밀번호가 평문으로 저장될 수 있었다

기존 구조를 완전히 버리지 않고, Django password hasher를 적용해 새 비밀번호는 해시로 저장하도록 변경했습니다. 기존 평문 비밀번호는 로그인 성공 시 자동으로 해시 값으로 전환되도록 했습니다.

### 문제 4. Q&A 흐름에서 예외 처리가 불명확했다

광범위한 예외 처리와 잘못된 필드 참조는 실제 오류를 숨기거나 런타임 실패를 만들 수 있습니다. 명시적인 객체 조회와 리다이렉트 흐름으로 정리하고, 질문 및 답변 작성 테스트를 추가했습니다.

이 프로젝트를 정리하면서 가장 크게 배운 점은 "포트폴리오는 결과 화면뿐 아니라 유지보수 과정도 설명할 수 있어야 한다"는 점입니다.

구체적으로는 다음을 배웠습니다.

- README와 Notion 문서는 서로 다른 목적을 가져야 함
  - README: 실행과 기능 확인 중심

## 12. Security / Confidentiality Note

현재 `WDB_2021_private` 저장소에서는 실제 시크릿 값과 로컬 데이터 파일을 Git 추적 대상에서 제거했습니다. 프로젝트는 환경 변수 기반으로 설정을 읽으며, 실제 값은 `hiseduplay/.env.local` 같은 ignored 파일이나 배포 환경 변수에만 보관하도록 정리했습니다.

다만 과거 public 저장소나 이전 커밋 기록에 노출된 적이 있는 값은 이미 유출된 것으로 간주해야 합니다. 따라서 실제 서비스에서 사용한 값이라면 반드시 폐기하고 새 값으로 교체해야 합니다.

포트폴리오에 공개할 때는 다음 원칙을 유지합니다.

- 실제 `SECRET_KEY` 공개 금지
- 실제 DB 계정 및 비밀번호 공개 금지
- 로컬 DB 파일 공개 금지
- `.env.local` 공개 금지
- 필요한 경우 `.env.example`만 공개

## 14. Interview Notes

### Q. 이 프로젝트를 왜 만들었나요?

사학 전공 배경을 웹 개발과 연결해보고 싶어서 만들었습니다. 한국사 콘텐츠를 단순히 문서로 보여주는 것이 아니라, 퀴즈와 Q&A를 함께 제공하는 작은 학습 웹사이트로 구성해보고자 했습니다.

### Q. 가장 중요하게 개선한 부분은 무엇인가요?

보안과 재현 가능성입니다. 기존 프로젝트에는 시크릿과 로컬 데이터가 섞여 있었고, 실행 방법도 명확하지 않았습니다. 이를 환경 변수 기반 설정, `.env.example`, 의존성 파일, 실행 문서로 정리했습니다. 또 평문 비밀번호 저장 위험을 줄이기 위해 Django password hasher를 적용했습니다.

### Q. 왜 기존 Member 모델을 유지했나요?

포트폴리오 유지보수 범위에서는 전체 인증 시스템을 갈아엎는 것보다, 기존 데이터 구조를 유지하면서 가장 위험한 비밀번호 저장 방식부터 개선하는 것이 현실적이라고 판단했습니다. 대신 새 가입자는 해시 저장을 적용하고, 기존 평문 비밀번호는 로그인 시 자동 업그레이드되도록 했습니다.

### Q. 테스트는 어떤 기준으로 추가했나요?

프로젝트에서 깨졌을 때 영향이 큰 흐름을 중심으로 추가했습니다. 인증, Q&A 작성, 답변 작성, 주요 페이지 렌더링이 핵심이었기 때문에 이 부분을 집중적으로 검증했습니다.

### Q. 이 프로젝트가 보여주는 강점은 무엇인가요?

단순 CRUD 프로젝트를 넘어, 과거 코드를 다시 읽고 보안 문제, 구조 문제, 실행 재현성 문제, 런타임 버그를 직접 개선했다는 점입니다. 새 기능만 만드는 것이 아니라 기존 프로젝트를 유지보수 가능한 상태로 바꾼 경험을 보여줄 수 있습니다.
